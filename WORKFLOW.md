# WORKFLOW — Stocks 项目开发工作流

> **每一条 Issue，无论大小，都必须走完这个完整流程。**
> 没有"小改动快车道"，没有"直接改"。
> 每一步都必须执行，不可跳过。

---

## 一、完整流程

```
用户提需求
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤1  建 Issue                            │
 │  ──────                                     │
 │  Hermes 在 GitHub 创建 Issue                 │
 │  · 标题带优先级前缀（P0/P1/P2）              │
 │  · 写清需求描述、现状、改动范围              │
 │  · 打标签（enhancement/bug/gui/exchange 等） │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤2  ARCH 拆解                            │
 │  ──────                                      │
 │  架构师在 Issue 评论中发布拆解方案：           │
 │  · 识别涉及哪些角色                           │
 │  · 每个角色分配具体任务（文件路径、改动内容）  │
 │  · 标注任务大小（小/中/大）                   │
 │  · 打角色标签（exchange/trader/strategy/gui/  │
 │    test/ci）                                  │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤3  建分支 + PR                          │
 │  ──────                                      │
 │  · git checkout -b feat/issue-<num>-<desc>   │
 │  · 先做第一个角色的最小改动（省去空 commit）   │
 │  · git push + 创建 PR                        │
 │  · PR body 贴拆解方案 + 每个子任务一个 checkbox│
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤4  各角色实现（链式执行）                │
 │  ──────                                      │
 │  按角色标签依次执行，每个角色：               │
 │  1. git pull 最新的 PR 分支                   │
 │  2. 仅修改自己角色目录下的文件                │
 │  3. 运行相关测试确认无回归                    │
 │  4. git commit + git push                    │
 │  5. 在 Issue 评论报告完成                     │
 │  6. 更新 PR body 中对应的 checkbox            │
 │                                              │
 │  如果本 tick 未超时（< 300s）：继续下一个角色  │
 │  如果超时：等下一个 cron tick                 │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤5  CI 自动运行                           │
 │  ──────                                      │
 │  GitHub Actions 运行 4 个 required checks    │
 │  · 自动化测试 (Automated Testing)             │
 │  · 代码检视 (Code Linting)                   │
 │  · GUI截图测试                                │
 │  · 自动化打包 (Automated Packaging)          │
 │  非 required checks（如 copilot-reviewer）    │
 │  失败不影响合并                               │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤6  LEAD 审查                             │
 │  ──────                                      │
 │  研发主管审查 PR diff：                       │
 │  · 检查文件所有权合规（不跨角色）             │
 │  · 检查代码质量                               │
 │  · 在 PR 评论发布审查结果（逐文件评审表）      │
 │                                              │
 │  然后检查 CI 状态：                           │
 │  · ✅ 全部通过 + 代码 OK → 打 needs-qa       │
 │  · ⏳ 仍在运行 → 保留 needs-review，下个 tick │
 │  · ❌ 有 required check 失败 → 仲裁（见下方）  │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤6a  CI 失败仲裁（按需）                  │
 │  ──────                                      │
 │  LEAD 判断失败原因：                          │
 │                                              │
 │  dev-bug → @对应角色修复，保留 needs-review  │
 │  test-needs-update → LEAD 直接修测试并 push   │
 │  flaky → 忽略，打 needs-qa 继续推进           │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤7  QA 验收                               │
 │  ──────                                      │
 │  · checkout PR 分支                          │
 │  · 运行全部测试：单元 + 集成 + GUI            │
 │  · 逐条核对 Issue 中的验收标准                │
 │  · 手动验证 UI 改动（如需）                   │
 │  · 发布 QA 报告到 PR 评论                     │
 └─────────────────────────────────────────────┘
  │
  ▼
 ┌─────────────────────────────────────────────┐
 │  步骤8  合并                                  │

  ▼
```

## 二、Issue 生命周期（标签流转）

每个 Issue 从创建到关闭经历以下标签状态：

| 阶段 | 标签 | 触发 |
|------|------|------|
| 新建 | `needs-triage` | Hermes 创建 Issue 时打上 |
| ARCH 拆解完成 | `triaged` + 角色标签 | ARCH 评论后打上 |
| 角色实现中 | `ai-in-progress` | 角色开始工作时打上 |
| 实现完成 | 角色标签移除 | 最后一个角色推完代码 |
| 待审查 | `needs-review` | 所有角色完成后打上 |
| 待验收 | `needs-qa` | LEAD Review 通过后打上 |
| 完成 | `ai-done` | QA 验收通过后打上 |

**链式执行时**（同一 tick 内连续跑多个角色）：
- 跳过所有标签 API 调用（标签无意义，角色就是自己）
- 最后一个角色推完后直接打 `needs-review`

## 三、角色分工

| 角色 | 代号 | 模型 | 负责目录 | 核心职责 |
|------|------|------|---------|---------|
| 产品经理 | PM | 普通 | — | 发现问题、提需求、评估优先级 |
| 架构师 | ARCH | **Pro** | — | 拆解需求、设计模块、分配角色任务 |
| 交易所 | EXCH | 普通 | `exchange/` | 行情数据、交易执行、账户管理 |
| 交易员 | TRADER | 普通 | `trader/` | 回测引擎、时间推进、买卖执行 |
| 策略算法 | STRAT | 普通 | `strategy/` | 策略逻辑、信号生成（不碰交易） |
| Web 前端 | GUI | 普通 | `gui/` | Flask 模板、JS 交互、UI 逻辑 |
| 集成测试 | ITEST | 普通 | `tests/unit/`, `tests/integration/` | 单元/集成测试，`test.yml` |
| GUI 测试 | GTEST | 普通 | `tests/guitests/` | GUI E2E 测试，`testgui.yml` |
| 研发主管 | LEAD | **Pro** | — | Review、仲裁 CI 失败、修测试 |
| 验收测试 | QA | — | — | 验收测试、确认需求、合并 PR |

### 每角色的实现流程

1. 拉取最新的 PR 分支：
   ```bash
   git fetch origin feat/issue-<num>-<desc>
   git checkout feat/issue-<num>-<desc>
   ```
2. **仅修改自己角色目录下的文件**（见 AGENTS.md §四 文件所有权）
3. 运行相关测试确认无回归：
   ```bash
   source .venv/bin/activate
   python -m pytest tests/ -q
   ```
4. Commit + push
5. 在 Issue 评论报告完成，更新 PR body 中的 checkbox

## 四、禁止行为

| ❌ | 说明 |
|----|------|
| 不在 Issue 外直接改代码 | 所有改动必须关联一个 Issue |
| 不跨角色提交代码 | EXCH 不能改 `gui/`，GUI 不能改 `exchange/` |
| 不直接推 main | 所有改动必须走 PR |
| 不在 stock-nightly/ 直接改 | 那是部署目录，只能 git pull |
| 不等 CI 就合并 | 必须 4/4 required checks 通过 |
| 不在 CI 失败时继续推进 | 必须先仲裁（dev-bug / test-needs-update / flaky） |
| 不跳过任何角色 | ARCH → 实现 → LEAD → QA → 合并，缺一不可 |
| 不跳过测试 | 每次 push 前自测，LEAD 审查时核验 |
| 不等 merge 确认就说"完成" | 只有 GitHub API 返回 `merged: true` 才算合并完成 |

## 五、LEAD Review 标准

| 检查项 | 要求 |
|--------|------|
| 文件所有权 | 每个改动文件必须在对应角色目录内 |
| 代码质量 | 无硬编码、无 `except:pass`、有适当 logging、类型提示 |
| 测试覆盖 | 新增功能有对应测试（ITEST/GTEST 负责） |
| CI 状态 | 所有 required checks 通过才可合并 |
| CI 失败仲裁 | dev-bug → @对应角色修复；test-needs-update → LEAD 修；flaky → 忽略继续 |

## 六、QA 验收标准

| 检查项 | 要求 |
|--------|------|
| 测试运行 | 单元 + 集成 + GUI 全部通过 |
| 无回归 | 已有测试全部通过 |
| 需求核对 | 逐条核对 Issue 中的验收标准 |
| CI 状态 | 合并前确认 required checks 全部通过 |
